Por Que os LLMs Precisam de RAG?

Antes de qualquer técnica: entendendo o problema que o RAG resolve

🎓

O Estudante que Decorou Tudo

Imagine um estudante brilhante que passou anos memorizando enciclopédias — cada livro, cada artigo publicado até 2023. Ele sabe de tudo. Mas o mundo não parou. Novos estudos surgem, fatos mudam, contextos evoluem. E o estudante? Ele não pode mais aprender. Seu conhecimento está congelado no momento em que terminou de estudar.

No mundo dos LLMs: o modelo é treinado até uma data de corte. Depois disso, tudo que acontece no mundo — novos regulamentos, resultados de pesquisa, documentos internos da sua empresa — é invisível para ele. Se você perguntar sobre algo recente ou privado, ele inventa (alucina) ou simplesmente não sabe. O RAG é o mecanismo que dá ao estudante acesso a uma mesa de pesquisa atualizada sempre que ele precisar responder.
Problema 1
Conhecimento congelado na data de treino — o modelo não sabe o que aconteceu depois
Problema 2
Dados privados e internos nunca estiveram no treino — a empresa não existe para o modelo
Problema 3
Alucinações: quando não sabe, inventa com confiança — e sem citar fonte
Bloco 0 · Fundação — Anatomia

O Que É um Sistema RAG, de Verdade?

3 componentes que trabalham juntos para dar memória atualizada ao LLM

🍳

A Cozinha do Chef

O Chef (o LLM) é talentoso, mas não tem ingredientes na cabeça. A Despensa (a base de documentos) tem tudo armazenado e organizado. A Comanda (a query do usuário) diz o que preparar. O trabalho do RAG é buscar os ingredientes certos na despensa, entregar ao chef, e ele cria o prato.

🗄️
Base de Documentos
PDFs, sites, wikis, DBs — qualquer fonte indexada em vetores para busca rápida
🔍
Retriever
Encontra os trechos mais relevantes para a pergunta do usuário usando embeddings
🤖
LLM Generator
Recebe a pergunta + os trechos recuperados e gera uma resposta fundamentada
O fluxo: Usuário pergunta → Retriever busca documentos relevantes → LLM lê pergunta + documentos → LLM gera resposta citando as fontes. Simples assim na teoria. O que muda entre os paradigmas é quão bem cada etapa é feita.
Bloco 0 · Fundação — Os 3 Paradigmas

A Evolução em Três Gerações

Cada geração surgiu para resolver os problemas da anterior — e todas ainda coexistem

Naive RAG · 2020
Recupera e gera, sem filtrar. Rápido, simples, mas erra muito em precisão.
Advanced RAG · 2022
Pré e pós-processamento. Reescreve queries, re-ordena docs, comprime contexto.
Modular RAG · 2023+
Módulos especializados plug-and-play. Cada componente pode ser trocado.
■ Bloco 1 · Naive RAG
📚

O Bibliotecário Ingênuo

Você entra na biblioteca e pergunta: "Qual é o melhor tratamento para diabetes tipo 2 em 2024?" O bibliotecário, sem pensar muito, vai até as prateleiras e te entrega 50 livros — alguns relevantes, muitos não. Ele não filtrou por data, não priorizou artigos clínicos, não verificou se o que ele pegou responde sua pergunta de fato. Agora você tem que ler tudo sozinho e montar a resposta.

No Naive RAG: quando o usuário pergunta, o sistema recupera um número fixo de documentos (ex.: top-5) usando busca vetorial simples, e entrega todos ao LLM sem avaliar se são relevantes, sem verificar se estão atualizados, sem comprimir o que é redundante. O LLM lê o "monte de livros" e tenta responder — às vezes bem, muitas vezes com alucinações nos gaps.
Bloco 1 · Naive RAG — Arquitetura

Como Funciona o Naive RAG por Dentro

Pipeline linear de 3 etapas — com os pontos onde tudo pode dar errado

1 · Indexação
Documentos são divididos em chunks fixos e transformados em vetores (embeddings). Armazenados em banco vetorial.
2 · Retrieval
Query do usuário é transformada em vetor. Busca os top-k documentos mais próximos por similaridade cosseno.
3 · Geração
LLM recebe [query + docs recuperados] como prompt e gera a resposta. Sem verificação adicional.
Bloco 1 · Naive RAG — Análise

Naive RAG: Quando Usar e Quando Evitar

Não existe abordagem ruim — existe abordagem errada para o contexto

Pontos Fortes
Implementação em horas — poucas dependências
Latência baixa — sem etapas extras de processamento
Custo operacional mínimo
Funciona bem para bases de dados pequenas e bem curadas
Ideal para protótipos e provas de conceito
Limitações Críticas
Baixa precisão — recupera documentos irrelevantes sem filtrar
Sem reavaliação — não verifica se o doc responde a query
Context window desperdiçado com informação redundante
Falha em queries ambíguas — não reformula a pergunta
Sem rastreabilidade — difícil saber por que errou
"O Naive RAG foi o ponto de partida que provou o conceito — mas suas limitações de precisão e recuperação foram o motor que impulsionou toda a evolução subsequente do campo."
Gao et al., RAG Survey LLMs, Tongji+Fudan, 2024
■ Bloco 2 · Advanced RAG
🔎

O Assistente de Pesquisa

Você pede a um assistente inteligente: "Preciso de informações sobre tratamentos de diabetes." Antes de ir à biblioteca, ele te faz perguntas: "Você quer tipo 1 ou tipo 2? Tratamentos medicamentosos ou comportamentais? Publicações dos últimos 3 anos?" Reformula sua busca, vai à biblioteca, pega 20 artigos — mas não te entrega todos. Ele lê, classifica por relevância, remove os redundantes e te dá um resumo estruturado dos 3 mais importantes.

No Advanced RAG: há duas fases adicionais. Pré-retrieval: a query é reformulada, expandida ou decomposta antes de buscar. Pós-retrieval: os documentos recuperados são re-ordenados (reranking), comprimidos, e só então entregues ao LLM. O resultado é um contexto muito mais limpo e preciso.
Bloco 2 · Advanced RAG — Pré-Retrieval

Pré-Retrieval: Perguntar Melhor para Achar Mais

A mesma pergunta, três formas de perguntar — e resultados completamente diferentes

Query Rewriting
Reformula a query original para maximizar relevância da busca vetorial. Remove ambiguidades.
Query Expansion
Gera múltiplas variações da query para cobrir diferentes ângulos e terminologias do domínio.
Query Decomposition
Decompõe uma pergunta complexa em sub-perguntas independentes respondidas separadamente.
Bloco 2 · Advanced RAG — Pós-Retrieval

Pós-Retrieval: Filtrar o Ruído, Entregar o Ouro

20 documentos entram no funil — 3 chegam ao LLM, prontos para usar

Reranking (Cross-Encoder)
Avalia cada par (query, documento) de forma mais sofisticada que a busca vetorial inicial. Re-ordena por relevância real. Modelos como Cohere Rerank ou BGE-Reranker.
Context Compression
Remove frases irrelevantes dos documentos selecionados. Mantém só o trecho que responde a query. Economiza tokens e reduz confusão no LLM.
Bloco 2 · Advanced RAG — Análise

Advanced RAG: Ganhos Reais, Custos Reais

Pontos Fortes
Precisão muito superior ao Naive — menos documentos irrelevantes
Redução significativa de alucinações pós-recuperação
Queries ambíguas tratadas antes de buscar
Context window usado com eficiência
Resultados mais rastreáveis e auditáveis
Trade-offs
Latência maior — cada etapa adiciona tempo
Mais componentes = mais pontos de falha
Reranker e compressor aumentam custo por consulta
Pipeline ainda é fixo — não se adapta ao tipo de pergunta
Ideal quando: qualidade importa mais que velocidade, base de documentos tem ruído ou inconsistência, ou o domínio é especializado (médico, jurídico, técnico).
■ Bloco 3 · Modular RAG
👨‍🍳

A Cozinha Profissional

Em um restaurante estrelado, não existe uma pessoa fazendo tudo. Há o garde-manger que prepara ingredientes frios, o saucier que faz os molhos, o chef de partie que monta o prato. Cada estação é especializada. Se o restaurante muda o cardápio, só troca a estação necessária — as outras continuam. E para pedidos diferentes, o fluxo entre estações muda dinamicamente.

No Modular RAG: cada função do pipeline — busca, reescrita, recuperação, reranking, leitura, memória — é um módulo independente que pode ser trocado, combinado ou reordenado. Um agente coordenador decide qual módulo chamar e em que ordem, baseado no tipo de pergunta. Frameworks como DSP e ITER-RETGEN tornam isso possível.
Bloco 3 · Modular RAG — Arquitetura

Módulos Plug-and-Play: Componha Seu Próprio RAG

6 módulos centrais — conectados em qualquer ordem segundo a necessidade

Search: busca em múltiplas fontes Rewrite: reformula queries Retrieve: recupera chunks Rerank: re-ordena por relevância Read: LLM lê e extrai Memory: armazena o que aprendeu
Bloco 3 · Modular RAG — Exemplos Reais

Modular RAG em Ação: DSP e ITER-RETGEN

Como o conceito de módulos virou sistemas concretos e poderosos

DSP · Demonstrate-Search-Predict
Usa poucos exemplos (few-shot) para que o LLM aprenda a decompor problemas e guiar a busca de forma autônoma. Cada módulo é otimizável independentemente.
Few-shot Self-guiding Modular otimizável
ITER-RETGEN · Recuperação Iterativa
A saída gerada pelo LLM na rodada anterior alimenta a próxima busca. O sistema aprende a refinar a recuperação com base no que já gerou — como um pesquisador que aprofunda o tema.
Iterativo Auto-refinamento Multi-hop
A virada do Modular RAG: o pipeline deixa de ser uma linha reta e passa a ser um grafo de decisão. Alguns nós são opcionais, alguns se repetem, e a ordem depende do contexto. Isso é o que os paradigmas S1 e S2 irão explorar a fundo.
■ Bloco 4 · Taxonomia das Famílias (Paper 02)

Onde Cada Pesquisador Inova: 4 Famílias

Sharma 2025 (ACM TOIS) organiza todo o campo em 4 famílias de inovação — cada uma com foco distinto

Retriever-Based
A inovação está em como buscar melhor. Query-centric (adaptar a busca à pergunta), Adaptive Retriever (o modelo decide quando buscar), Granularity (qual nível de chunking usar).
Ex.: HyDE · FLARE · Adaptive-RAG
Generator-Based
A inovação está no gerador. Faithfulness (forçar o LLM a citar), Compression (comprimir contexto antes de gerar), Guided Generation (tokens que guiam o output).
Ex.: FILCO · SEER · Self-RAG
Hybrid Systems
A inovação está na coordenação entre retriever e generator. Multi-round (vai e volta), Dynamic Triggering (decide quando recuperar), Utility-driven (pondera utilidade de cada doc).
Ex.: CRAG · Speculative RAG · ITER-RETGEN
Robustness-Oriented
A inovação está em não errar sob pressão. Noise-Adaptive (funciona com docs ruidosos), Hallucination detection, Adversarial (resistência a ataques na base de docs).
Ex.: FILCO · Noise-RAG · Adversarial eval
Bloco 4 · Taxonomia — Robustez
🪞

A Sala dos Espelhos

Imagine que seu assistente de pesquisa foi trabalhar em uma sala cheia de espelhos distorcidos. Alguns documentos são reais. Outros são cópias com partes alteradas. Outros foram inseridos por alguém mal-intencionado para enganar. E o assistente não sabe distinguir o espelho da janela. Quanto mais ele confia no que vê, mais distorcida fica a resposta final.

No mundo RAG: bases de documentos reais têm ruído — informações desatualizadas, documentos contraditórios, dados de baixa qualidade. Ataques adversariais podem injetar documentos falsos. Robustez é a capacidade do sistema de reconhecer esse ruído e não se deixar enganar por ele.
Ruído
Documentos desatualizados, incompletos ou tangencialmente relevantes que poluem o contexto
Alucinação
O LLM ignora os documentos recuperados e "inventa" detalhes que não estão na fonte
Adversarial
Documentos injetados intencionalmente para manipular a resposta do sistema RAG

■ Bloco 5 · Engenharia de Produção (Paper 03)
🏭

A Planta Industrial

Uma fábrica que produz remédios não pode funcionar como um laboratório experimental. Cada etapa de produção tem estações de controle de qualidade: como os ingredientes são processados (Retrieval), como são combinados (Fusion), em quais formatos podem chegar (Modality), se a linha se adapta automaticamente (Adaptivity) e se há inspeção de segurança no final (Trust Layer). Pular qualquer estação compromete o produto inteiro.

No Engineering the RAG Stack (IBM/AI Alliance, 2025): uma revisão sistemática de 2018-2025 identificou que sistemas RAG de produção falham principalmente quando ignoram uma das 5 dimensões arquiteturais. Não são opcionais — são os 5 pilares que sustentam confiabilidade em escala.

🤐

Bloco 6 · A Fórmula que Unifica

A Equação que Define o RAG

Uma fórmula simples que esconde toda a complexidade — e conecta os 3 paradigmas

P(y | x) = Σᵢ Pret(zᵢ | x) · Pgen(y | x, zᵢ)
Resposta y dada query x = soma sobre todos os docs zᵢ de (relevância do doc × probabilidade de gerar y com aquele doc)
🗳️

Metáfora: O Sistema de Votação

Imagine uma eleição onde cada documento recuperado vota na resposta. Mas não é um voto igual — documentos mais relevantes para a query têm mais peso (Pret). E cada documento "suporta" uma resposta com uma certa probabilidade (Pgen). A resposta final é a que recebe mais votos ponderados. O RAG é uma democracia dos documentos.

Pret(zᵢ|x)
Quão relevante é o documento para a query
Pgen(y|x,zᵢ)
Probabilidade de gerar y dado query + documento
Σᵢ
Integração sobre todos os documentos recuperados
■ Bloco 7 · Sistema 1 vs Sistema 2 (Paper 05)
🚗

Kahneman no Volante

Daniel Kahneman, Prêmio Nobel de Economia, descreveu dois modos de pensar. O Sistema 1 é o piloto automático: você dirige pela rota de sempre sem pensar, reage rápido, gasta pouca energia. O Sistema 2 é a deliberação: quando aparece um desvio inesperado no meio da estrada, você para de dirigir no automático, pensa com cuidado, avalia opções e decide conscientemente.

No RAG (Liang et al., BUPT+TUM, 2025): Sistema 1 = Predefined Reasoning (pipelines fixos, rápidos, módulos pré-definidos). Sistema 2 = Agentic Reasoning (o modelo decide dinamicamente o que fazer, quando buscar, quantas vezes iterar). 2025 é o ano em que o S2 virou o foco central do campo.
Bloco 7 · Sistema 1 · Predefined Reasoning

Sistema 1: Rápido, Controlado, Modulado

4 arquiteturas de pipelines predefinidos — cada uma com um padrão diferente de raciocínio

Route-Based
RAGate, Self-Route
Primeiro decide se recuperar. Uma porta de entrada classifica a query e redireciona para o caminho certo: RAG completo, só LLM, ou LLM com contexto curto.
Metáfora: Recepcionista que encaminha ao departamento certo
Loop-Based
Self-RAG, CRAG, IRCoT
Recupera, avalia a qualidade, decide se precisa recuperar de novo. O loop continua até a resposta ser satisfatória ou o limite de iterações ser atingido.
Metáfora: Editor que revisa o próprio texto antes de publicar
Tree-Based
RAPTOR, MCTS-RAG, ToT
Expande o espaço de busca como uma árvore. Cada nó é um estado parcial. Usa MCTS (Monte Carlo Tree Search) para explorar e podar caminhos de raciocínio.
Metáfora: Enxadrista avaliando jogadas a múltiplos movimentos
Hybrid-Modular
Adaptive-RAG, Flex-RAG
Combina múltiplos padrões. Classifica a query e escolhe dinamicamente entre single-hop, multi-hop ou pipeline completo. Melhor custo-benefício.
Metáfora: Piloto que alterna entre manual e automático segundo o tempo
Bloco 7 · Sistema 2 · Agentic Reasoning

Sistema 2: O Agente que Decide por Si Mesmo

Dois caminhos para chegar ao raciocínio autônomo: via prompt ou via treinamento

Prompt-Based (sem retreinar)
ReAct: Thought → Action → Observation em loop. O modelo raciocina antes de agir e observa o resultado antes do próximo passo.
Self-Ask: Decompõe a pergunta em sub-perguntas e responde cada uma antes de responder a original.
Search-O1: Combina chain-of-thought com busca web em tempo real para problemas de raciocínio complexo.
Vantagem: funciona com qualquer LLM, sem necessidade de fine-tuning.
Training-Based (retreinado)
Search-R1: Fine-tuning com RL (GRPO) — o modelo aprende quando e como buscar como uma habilidade emergente.
DeepRetrieval: Treina o modelo para formular queries de busca eficazes, não só para responder.
DeepResearcher: Agente multi-step com browsing real, treino GRPO + F1 reward em ambiente web real.
Vantagem: comportamento mais consistente e emergência de estratégias novas de busca.
2025: o ano agêntico. 55,9% dos trabalhos de Reasoning RAG são prompt-based (S2 sem treino) — sinal de que o campo está em explosão. O S2 training-based tende a superar o prompt-based, mas exige mais recursos.

■ Bloco 8 · Self-RAG (Paper 06 · ICLR 2024)
🔍

O Detetive

Sherlock Holmes nunca aceita a primeira evidência passivamente. Quando encontra uma pista, ele pergunta: "Esta evidência é relevante para o caso?" Se sim: "Ela sustenta minha teoria ou a contradiz?" E antes de concluir: "Esta conclusão é útil para o cliente?" Ele faz isso para cada pista, em cada passo — não no final, não às cegas.

No Self-RAG (Asai et al., UW+AI2+IBM, ICLR 2024): o modelo aprende a inserir reflection tokens — marcadores especiais que interrompem a geração e fazem o modelo se perguntar exatamente as mesmas questões do detetive. O resultado: recuperação on-demand (só quando necessário) e auto-avaliação contínua da qualidade do que está gerando.

■ Bloco 9 · Speculative RAG (Paper 07 · ICLR 2025)
⚡

O Estagiário Rápido + O Sênior Criterioso

Em um escritório de advocacia, quando chega um caso urgente, o sócio sênior não pesquisa tudo do zero. Ele manda 5 estagiários em paralelo, cada um com um ângulo diferente do caso. Em 10 minutos, tem 5 rascunhos. O sócio sênior lê os 5, avalia a solidez de cada argumento, e assina o melhor — ou combina partes. Resultado: qualidade de sênior, velocidade de estagiário.

No Speculative RAG (Wang et al., UCSD+Google, ICLR 2025): o Drafter (Mistral-7B, especialista pequeno) gera m rascunhos em paralelo, cada um com k documentos de uma perspectiva diferente (via K-Means clustering). O Verifier (Mixtral-8x7B, generalista maior) escolhe o melhor pelo score composto ρⱼ. Resultado: +12,97% precisão, -50,83% latência.
Bloco 9 · Speculative RAG — K-Means Sampling

Por Que K-Means? A Magia das Perspectivas Diversas

Em vez de pegar os top-5 mais similares (redundantes), divide os docs em 5 clusters — cada um com uma perspectiva diferente

O problema do naive top-k: se você pega os 5 documentos mais similares à query, provavelmente pega 5 documentos quase idênticos — e o LLM não aprende nada novo do 2º ao 5º. O K-Means garante diversidade de perspectiva: cada cluster representa um ângulo diferente sobre o mesmo tema.

Bloco 10 · Para Onde o Campo Se Move

4 Fronteiras Que Ainda Ninguém Resolveu

Oportunidades abertas identificadas pelos 7 papers — onde as próximas grandes contribuições virão

RAG Agêntico com RL (Sistema 2 Training)
Search-R1 e DeepResearcher abriram o caminho — mas treinamento em ambientes reais ainda é instável e caro. A próxima fronteira: multi-agente coordenado com RL em dados proprietários.
RAG Multi-modal Nativo
Tabelas, gráficos, imagens e código como cidadãos de primeira classe — não convertidos para texto antes. O desafio: embeddings multi-modais sem degradação semântica.
Privacy-Preserving RAG
Federated retrieval, differential privacy em embeddings, acesso granular por perfil. Ainda sem solução padronizada — e sem ela, RAG corporativo tem teto baixo.
Avaliação Padronizada
Engineering the RAG Stack (IBM) é enfático: sem benchmarks unificados e métricas consensuais, comparações entre sistemas são impossíveis. RAGAS é dominante, mas não suficiente para todos os cenários.

Qual RAG Usar? O Mapa de Trilhas

A escolha certa depende de 3 variáveis: orçamento de latência, complexidade da tarefa e necessidade de auditoria

Quando usar Naive RAG
Protótipo rápido · Base pequena e bem curada · Baixo orçamento · Latência crítica · FAQ simples
Quando usar Advanced RAG
Domínio especializado · Qualidade > velocidade · Base ruidosa · Queries ambíguas · Resumo técnico
Quando usar Self-RAG
Rastreabilidade necessária · Alucinação é crítica · Multi-documentos · Auditoria por passo · Saúde/jurídico
Quando usar Speculative RAG
Alta escala · Latência + qualidade · Base grande com perspectivas diversas · Produção com SLA de latência
