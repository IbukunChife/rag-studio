Metáfora · O que é Reranking
O Concurso de Segundo Turno

Em uma eleição com 100 candidatos, o primeiro turno elimina os menos votados rapidamente usando uma regra simples. O segundo turno analisa os finalistas com muito mais profundidade — quem é mais capacitado para o cargo específico? O Reranking em RAG funciona exatamente assim: o retriever faz o "primeiro turno" (encontra os K documentos semanticamente próximos), e o reranker faz o "segundo turno" — analisa cada par (pergunta + documento) para identificar quais realmente respondem bem à intenção do usuário.

Sem Reranking — Primeiro Turno apenas

Documentos rankeados por similaridade de embedding. Rápido, mas frequentemente retorna textos similares que não respondem diretamente à pergunta — "Lost in the Middle" afeta o LLM.

Com Reranking — Segundo Turno fino

Cada documento é avaliado em conjunto com a query — quão bem ele responde a esta pergunta específica? Mais lento, mas o LLM recebe os documentos mais relevantes primeiro.

ARAGOG · Mecanismo

Por que HyDE Ganha e MMR Falha

A intuição por trás dos resultados surpreendentes do ARAGOG

HyDE — Por que ganha

Uma query de usuário é curta e ambígua. A resposta hipotética do LLM é longa e contextualizada — ocupa um espaço de embedding muito mais próximo dos documentos relevantes. É preencher o gap de vocabulário entre perguntas e respostas.

MMR — Por que falha

MMR diversifica os documentos para evitar redundância. Mas em RAG, o LLM precisa de múltiplos fragmentos do mesmo tópico para confirmar evidências. Diversidade em excesso dilui o sinal relevante.

LCR · Motivação

Rerankers Existentes: Três Limitações

Dalian University of Technology + Tencent · arXiv Fev 2026

①
Treinamento Específico Obrigatório

Rerankers supervisionados (cross-encoders fine-tuned) dependem de dados de treinamento específicos do domínio. Generalização ruim para novos domínios. Alto custo computacional de treinamento.

②
Rerankers LLM Ignoram Confiança

Rerankers baseados em LLM usam relevância ou similaridade como sinal. Mas subutilizam a confiança intrínseca do LLM — um sinal poderoso derivado do processo generativo que se correlaciona fortemente com relevância do documento.

③
NDCG@5 < 0.8 em BEIR

Mesmo state-of-the-art rerankers em benchmarks como BEIR frequentemente ficam abaixo de 0.8 de NDCG — indicando espaço significativo para melhorias sem retreinamento.

Metáfora · LCR
O Perito de Black-Box

Imagine um perito especialista que avalia documentos para responder perguntas. Você não tem acesso ao interior da mente dele — apenas observa quão confiante ele soa nas respostas que dá. Quando ele responde várias vezes de forma muito consistente, você sabe que está certo. Quando ele hesita e muda de resposta, o documento provavelmente não tem a informação. O LCR (LLM-Confidence Reranker) usa exatamente isso: mede a confiança do LLM ao gerar respostas baseadas em cada documento candidato. Alta confiança = documento relevante.

Black-Box

Sem acesso a parâmetros internos ou probabilidades brutas — apenas saídas do LLM

Amostragem

Gera N respostas para cada par (query, documento) via multinomial sampling

MSCP

Maximum Semantic Cluster Proportion — o cluster semântico dominante indica confiança

LCR · MSCP

Confiança Medida por Agrupamento Semântico

Maximum Semantic Cluster Proportion — quanto maior o cluster dominante, maior a confiança

Alto MSCP (0.8+)

LLM sempre diz o mesmo → documento altamente relevante

MSCP Médio (0.5–0.8)

Alguma variação → relevância moderada, ordena conforme threshold

Baixo MSCP (<0.5)

LLM hesita muito → documento provavelmente irrelevante

LCR · Algoritmo

Duas Fases: Avaliar + Ordenar

LLMs com 7-9B parâmetros — sem treinamento, sem acesso a pesos internos

Fase 1 — Confidence Assessment
  1. Para cada doc candidato: (query + doc) → prompt ao LLM
  2. Gera N amostras via multinomial sampling
  3. Agrupa semanticamente as N respostas
  4. MSCP = tamanho do maior cluster / N
Fase 2 — Binning + Multi-Level Sort
  1. Query confidence threshold: query muito específica? Preserve ranking original
  2. Document confidence threshold: divide em bins alto/médio/baixo
  3. Ordena por bin de confiança de forma descendente
  4. Alta confiança de query → mantém original (robustez)

Robustez: Para queries de alta confiança, o LCR preserva o ranking original — evita degradação quando o retriever já recuperou os documentos certos.

Neural Pipelines · Motivação

RAG em Grafos de Conhecimento E-Commerce

Abertay University, UK · arXiv Dez 2025 — Amazon STaRK SKB (Semi-structured Knowledge Base)

E-commerce é um domínio especialmente desafiador para RAG: catálogos com milhões de produtos, atributos constantemente atualizados (preço, estoque, avaliações), queries que vão de precisas ("código do produto X") a vagas ("produto de limpeza para tecido fine"). A escala e natureza semi-estruturada dos grafos de conhecimento (Knowledge Graphs) exige estratégias especializadas de retrieval + reranking.

Amazon STaRK SKB
  • Semi-structured Knowledge Base proprietária
  • Dados detalhados de atributos de produtos
  • Inter-product relationships (edges do grafo)
  • Benchmark ground truth de respostas de nodes
O Problema Específico
  • Multi-hop reasoning entre entidades do grafo
  • Retrieval semântico insuficiente para relações explícitas
  • Cross-encoder refina precisão mas não via grafo
  • Retrieval esparso (BM25) capta lexical mas perde semântica
Metáfora · Neural Pipelines
O Catálogo de Produtos com Júri

Uma loja online com 1 bilhão de produtos recebe a pergunta: "Qual produto de limpeza remove manchas de café de tecido fine sem danificar fibras naturais?". Um motor de busca comum retorna 50 produtos com termos parecidos. Mas um júri de especialistas (cross-encoder) avalia cada par (pergunta + produto) individualmente e determina qual realmente atende. Os pipelines FRWSR, FRMR e BARMR são três formatos diferentes desse júri — alguns usam especialistas densos (FAISS), outros usam especialistas lexicais (BM25 com arestas do grafo).

FRWSR
Dense + Webis

FAISS-HNSW + Webis Set-Encoder reranking

FRMR
Dense + MS MARCO

FAISS-HNSW + MS MARCO MiniLM cross-encoder

BARMR
Sparse + Grafo

BM25 + edge augmentation SKB + MS MARCO

Neural Pipelines · Arquitetura

Três Pipelines, Uma Comparação

Retriever + Reranker como 2 estágios — cada pipeline difere em ambos os componentes

Todos os três pipelines usam E5-Large embeddings para FAISS (nos dois pipelines densos). O BARMR usa arestas do grafo SKB para enriquecer o retrieval BM25 com relacionamentos entre produtos.

Neural Pipelines · Arquitetura

Bi-Encoder vs. Cross-Encoder

O trade-off fundamental entre velocidade e precisão em reranking

Bi-Encoder — Retrieval Eficiente
  • Query e documento codificados separadamente
  • Vetores pré-computados para documentos
  • Busca via similaridade de cosseno
  • O(1) por par — ideal para escala
Cross-Encoder — Reranking Preciso
  • Query + documento codificados em conjunto
  • Attention full entre tokens dos dois textos
  • Score de relevância direta, não apenas cosseno
  • O(n) por consulta — só viável pós-retrieval

Módulo 11 · Aprendizados

Quatro Lições de Reranking

①
HyDE > MMR para retrieval precision — a hipótese preenche o gap de vocabulário

ARAGOG prova: gerar uma resposta hipotética antes de buscar alinha melhor o espaço de embedding. MMR diversifica quando o LLM precisa de convergência.

②
Confiança do LLM é um sinal de relevância — use-a sem treinamento

LCR demonstra +20.6% NDCG@5 usando apenas amostragem e agrupamento semântico de respostas LLM. A confiança correlaciona com relevância — hipótese validada experimentalmente.

③
Cross-encoder é essencial para domínios estruturados como e-commerce

Neural Pipelines: a análise conjunta (query + documento) captura relações semânticas finas que similaridade de cosseno isolada perde. +20.4% Hit@1 valida a arquitetura 2-stage em KG.

④
Nenhum reranker é universal — avalie no seu domínio com suas métricas

ARAGOG, LCR e Neural Pipelines juntos mostram: cada técnica tem contexto ótimo. Cohere falha no arXiv mas pode funcionar em outros domínios. Meça retrieval precision E answer similarity sempre.
