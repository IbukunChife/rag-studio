Metáfora · O que é Reranking
🗳️
O Concurso de Segundo Turno

Em uma eleição com 100 candidatos, o primeiro turno elimina os menos votados com uma regra simples. O segundo turno analisa os finalistas com muito mais profundidade — quem é mais capacitado para esse cargo específico? Reranking em RAG funciona assim: o retriever faz o "primeiro turno" (encontra os K documentos semanticamente próximos), e o reranker faz o "segundo turno" — avalia cada par (pergunta + documento) para identificar quais realmente respondem à intenção do usuário.

Sem Reranking (só 1º turno)

Documentos rankeados por similaridade de embedding. Rápido, mas frequentemente retorna textos similares que não respondem diretamente — "Lost in the Middle" afeta o LLM.

Com Reranking (2º turno fino)

Cada documento avaliado em conjunto com a query — quão bem ele responde a esta pergunta específica? Mais lento, mas o LLM recebe os documentos mais relevantes primeiro.

ARAGOG · Mecanismo

HyDE Preenche o Gap, MMR Introduz Ruído

A intuição por trás dos resultados contraintuitivos

HyDE — O gap de vocabulário

Query de usuário é curta e ambígua. A resposta hipotética do LLM é longa e contextualizada — muito mais próxima do espaço de embedding dos documentos relevantes. HyDE converte uma pergunta no idioma da resposta.

MMR — Diversidade que dilui sinal

MMR diversifica para evitar redundância. Mas o LLM precisa de múltiplos fragmentos do mesmo tópico para confirmar evidências. Diversidade em excesso dilui o sinal relevante e envia ruído para o contexto.

LCR · Motivação

Rerankers Atuais: Três Limitações

Dalian University of Technology + Tencent · arXiv Fev 2026

①
Fine-Tuning obrigatório

Cross-encoders fine-tuned dependem de dados de treinamento específicos do domínio. Generalização ruim para novos domínios. Alto custo de treinamento e manutenção.

②
Confiança intrínseca do LLM ignorada

Rerankers LLM usam relevância ou similaridade, mas subutilizam a confiança intrínseca gerada pelo processo generativo — um sinal que correlaciona fortemente com relevância do documento.

③
NDCG@5 < 0,8 nos benchmarks BEIR

Mesmo state-of-the-art rerankers ficam abaixo de 0,8 NDCG@5 em BEIR — espaço significativo para melhorias sem retreinamento.

Metáfora · LCR
🧠
O Perito de Black-Box

Imagine um perito especialista que avalia documentos para responder perguntas. Você não tem acesso ao interior da mente dele — apenas observa quão confiante ele soa nas respostas. Quando ele responde várias vezes de forma muito consistente, está certo. Quando hesita e muda de resposta, o documento provavelmente não tem a informação. O LCR (LLM-Confidence Reranker) usa exatamente isso: mede a confiança do LLM ao gerar respostas baseadas em cada documento candidato. Alta confiança = documento relevante.

Black-Box

Zero acesso a parâmetros internos — apenas saídas do LLM

Amostragem

Gera N respostas para cada par (query, doc) via multinomial sampling

MSCP

Maximum Semantic Cluster Proportion — cluster dominante = confiança

LCR · MSCP

Confiança via Votação de Respostas

N amostras → agrupamento semântico → cluster dominante define score de confiança

MSCP alto (0,8+)

LLM converge → doc altamente relevante → sobe no ranking

MSCP médio (0,5–0,8)

Alguma variação → relevância moderada → mantém posição

MSCP baixo (<0,5)

LLM hesita → doc provavelmente irrelevante → desce no ranking

LCR · Algoritmo

Duas Fases: Avaliar + Ordenar com Robustez

LLMs 7–9B parâmetros · sem treinamento · sem acesso a pesos internos

Fase 1 — Confidence Assessment
  1. Para cada doc candidato: (query + doc) → prompt ao LLM
  2. Gera N amostras via multinomial sampling
  3. Agrupa semanticamente as N respostas
  4. MSCP = tamanho do maior cluster / N
Fase 2 — Binning + Multi-Level Sort
  1. Query confidence threshold: query muito específica? Preserve ranking original
  2. Document bins: alto/médio/baixo MSCP
  3. Ordena por bin de confiança descendente
  4. Alta confiança de query → mantém original (robustez)

Robustez por design: Para queries de alta confiança, LCR preserva o ranking original — evita degradação quando o retriever já recuperou os documentos certos. Não piora o que já está bom.

Neural Pipelines · Fundação

Bi-Encoder vs. Cross-Encoder

O trade-off fundamental entre velocidade e precisão

Bi-Encoder — Retrieval em Escala

Query e documento codificados separadamente. Vetores pré-computados para documentos. Busca via cosine similarity. O(1) por par — viável para milhões de docs.

Cross-Encoder — Reranking Preciso

Query + documento codificados em conjunto. Full attention entre tokens dos dois textos. Score de relevância direta. O(K) por consulta — só viável após retrieval filtrar para K docs.

Tensão Conceitual · O Debate

Training-Free vs. Fine-Tuned: Qual Vence?

🧠 LCR — Training-Free
✓ +20,6% NDCG@5 vs. cross-encoders fine-tuned
✓ Zero custo de treinamento — novo domínio imediatamente
✓ Robustez: não degrada queries de alta confiança
✗ Mais lento: N inferences por documento candidato
✗ Custo de inference cresce com N amostras e K docs
🔧 Cross-Encoder Fine-Tuned
✓ Inferência única por par — muito mais rápido que LCR
✓ Especializado no domínio de treinamento — NDCG alto no domínio
✓ Neural Pipelines: +20,4% Hit@1 com cross-encoder MS MARCO
✗ Generalização ruim para domínios fora do treinamento
✗ ARAGOG: Cohere Rerank sem ganho em arXiv — evidência de fragililidade

Veredicto: LCR é superior em generalização e performance absoluta. Cross-encoder fine-tuned é superior em velocidade e domínio específico com dados de treinamento. A escolha depende do orçamento de latência vs. custo de dados rotulados — não há vencedor universal.

Módulo 11 · Princípios Destilados

O Júri Certo para Cada Eleição

Lembram do segundo turno da eleição? ARAGOG descobriu quem são os melhores juízes — e que alguns famosos (Cohere, MMR) não rendem o esperado. LCR descobriu que o juiz pode usar sua própria confiança como critério, sem precisar de treinamento específico — e supera os treinados. Neural Pipelines trouxe o grafo de relacionamentos ao júri, para que os produtos concorrentes conheçam seus vizinhos no catálogo. A eleição é sempre em dois turnos. O segredo é escolher o júri certo para o segundo.

P1

Diversidade de query não é o mesmo que qualidade — MMR e Multi-query introduzem ruído. HyDE preenche o gap de vocabulário sem adicionar ruído.

P2

Confiança do LLM é sinal de relevância — mensurável sem treinamento — LCR valida que MSCP (cluster semântico dominante) supera cross-encoders fine-tuned em NDCG@5.

P3

Cross-encoder em grafos precisa de arestas — BM25 com edge augmentation (BARMR) supera FAISS sem augmentation em KGs de e-commerce.

P4

Nenhum reranker é universal — Cohere falha em arXiv, MSCP excelente em BEIR. Avalie com as suas métricas, no seu corpus, com o seu baseline.
