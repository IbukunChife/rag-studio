O Problema Central

Tabelas Não São Parágrafos

O RAG clássico lineariza tabelas em texto — e perde exatamente a estrutura relacional que dá significado aos dados

Perda de Estrutura

Linearizar tabelas em Markdown destrói relações entre linhas e colunas — raciocínio multi-hop fica inviável.

Sem Visão Global

Chunking fragmenta a tabela. "Quantos no total?" exige a tabela inteira — não um trecho de contexto.

Ruído Conversacional

Em chatbots, o conhecimento útil está enterrado em dezenas de turnos irrelevantes e saudações.

A virada: tratar a tabela como uma unidade indivisível de raciocínio — via SQL simbólico, grafos semânticos ou índices hierárquicos que preservam a relação linha ↔ coluna. Todos os 5 papers desta semana chegaram à mesma conclusão.

Metáfora · T-RAG (Illinois/Meta/IBM)
📚
Biblioteca com Índice em Três Dimensões

Imagine uma biblioteca gigante com 57 mil livros. Uma abordagem ingênua lista todos os títulos em ordem alfabética. O T-RAG é como um bibliotecário que organiza os livros de três formas simultâneas: pelo assunto (semântica via Sentence Transformer), pelo tamanho e formato (estrutura via spaCy POS) e pela frequência de consulta (heurística via TF-IDF). Quando você faz uma pergunta, ele vai direto ao cluster certo nas três dimensões — e então afina com busca de subgrafo local. Nenhum livro irrelevante é sequer tocado.

Hipergrafos Heterogêneos

Tabelas são nós do grafo; hiperarestas agrupam tabelas similares em 3 tipos: semântica, estrutural, heurística. KMeans tridimensional gera clusters independentes.

Busca Grossa → Fina

Fase 1: seleciona cluster por dimensão. Fase 2: busca subgrafo local. Fase 3: graph-aware prompting para LLM — raciocínio cross-table explícito.

Animação Conceitual · O Dilema Fundamental

Linearizar em Texto vs. Preservar Estrutura

A mesma tabela percorre dois caminhos — um perde estrutura, o outro a preserva. A pergunta chega ao mesmo dado, mas o resultado é completamente diferente.

Caminho A: Linearização

Tabela → texto Markdown → chunk → embed. A célula "12.11" vira texto sem saber que é "Receita, Outubro, Segmento X". Queries de aggregação e cross-referência falham.

Caminho B: Estruturação

Tabela → SAT-Graph/SQL/Hipergrafo. Cada valor carrega coordenadas (linha, coluna, contexto temporal). Query "receita Outubro Seg-X" vai exatamente à célula certa.

Metáfora · FT-RAG (Huawei · KDD 2026)
🔍
A Lupa que Lê Cada Célula

Imagine um analista financeiro que precisa encontrar a receita doméstica do Segmento X no terceiro trimestre de 2024 num relatório de 200 páginas. O RAG convencional escaneia o texto em blocos — como folhar rápido. O FT-RAG é como uma lupa especial: ela não lê "a tabela", ela lê cada célula individualmente, sabendo exatamente que "12.11" está na linha Domestic, coluna October, Segmento X, Tabela 2, Empresa A — com toda a cadeia de contexto preservada em coordenadas precisas. O SAT-Graph guarda sujeito, atributo e tempo como sub-árvores separadas, unindo o que a tabela 2D separa.

Subject Sub-tree (Ts)

Inferido via LLM sobre contexto global da Cell Group. Reconstrói hierarquia de sujeitos — "Laptop" aninhado sob "Hardware".

Temporal Sub-tree (Tt)

Via parsing de padrões. Organiza intervalos cronológicos — October, November, December como scopes de validade do dado.

Metáfora · TableRAG (Huawei) · NeuSym-RAG (SJTU)
🕵️
O Detetive que Interroga o Banco de Dados

Um detetive ruim faz perguntas vagas. Um detetive expert sabe exatamente o que perguntar: "Me dê os suspeitos que estavam na cena entre 21h e 23h, ordenados pela hora de saída." O TableRAG usa SQL como linguagem de interrogação precisa — determinístico, sem overhead de Python, indivisível. O NeuSym-RAG vai além: é o detetive bilíngue que decide entre SQL (preciso) e embedding semântico (fuzzy) dependendo da pergunta — e às vezes usa os dois em sequência via framework ReAct.

TableRAG: SQL + Vetorial Híbrido

Dois corpora paralelos: KB textual (chunks) + schema MySQL (tabelas). Subquery vai para SQL quando o chunk é de tabela, para retrieval quando é texto. +17.3% HeteQA.

NeuSym-RAG: 5 Ações ReAct

RetrieveVS · RetrieveDB · ViewImage · CalculateExpr · GenerateAnswer. ViewImage "vê" o que o OCR não parseia. CalculateExpr evita alucinação matemática. +17.3% AirQA-Real.

Metáfora · SA-RAG (UCLA/NEC)
🏥
A Triagem do Pronto-Socorro

Um pronto-socorro recebe pacientes em estado caótico: gritos, dores de todo tipo, informações contraditórias. A triagem não tenta processar tudo — ela converte o caos em uma ficha estruturada: nome, sintoma principal, urgência, histórico relevante. O SA-RAG faz o mesmo com dados conversacionais: cada conversa barulhenta de chatbot é convertida em uma tabela limpa, com metadados de qualidade validados via função L(M), eliminando turnos irrelevantes e normalizando o conhecimento útil antes de qualquer busca. Dados ruidosos exigem normalização antes do indexamento — lixo estruturado ainda é lixo.

Tensão Conceitual · O Debate

Grafo Celular vs. SQL Simbólico

🔍 FT-RAG (SAT-Graph Celular)

Tese: a célula é a unidade mínima de raciocínio tabular. Preservar sujeito, atributo e tempo em sub-árvores explícitas permite raciocínio cross-dimensional que SQL não captura: "qual empresa teve crescimento de receita doméstica em todos os trimestres de 2024?" exige travessia temporal + hierarquia de sujeito.

+62.2% Exact Value · tabelas nested e temporais
🕵️ TableRAG (SQL Determinístico)

Tese: SQL é a linguagem mais expressiva para raciocínio tabular — e já foi testado por décadas. Para documentos heterogêneos (texto + tabela), a dualidade vetorial/SQL permite cobertura completa sem construir um grafo interno. Funciona em tabelas regulares com schema limpo sem overhead de construção do SAT-Graph.

SOTA HeteQA · docs heterogêneos · schema regular

Síntese: FT-RAG vence em tabelas financeiras/científicas com dados temporais aninhados — onde a hierarquia importa e os valores têm validade temporal. TableRAG vence em documentos heterogêneos texto+tabela com schemas regulares — onde SQL é mais expressivo e eficiente que grafo. Use FT-RAG quando a estrutura interna da tabela é complexa; TableRAG quando o desafio é a fusão texto+tabela.

Módulo 16 · Princípios Destilados

Os 4 Especialistas em Raciocínio Tabular Coordenados

Lembram dos 4 especialistas? A Biblioteca Tridimensional (T-RAG) uniu semântica, estrutura e heurística num hipergrafo — e chegou a 89% de acurácia em 57 mil livros que nenhum índice flat jamais tinha organizado bem. A Lupa Celular (FT-RAG) revelou que cada célula precisa saber o seu sujeito, atributo e tempo — o SAT-Graph deu +62% de exatidão ao que antes era um texto sem coordenadas. O Detetive SQL (TableRAG + NeuSym) provou que a linguagem certa para tabelas é SQL — determinístico, expressivo, sem overhead de embedding. A Triagem (SA-RAG) lembrou que nenhuma estrutura conserta lixo não normalizado: normalize antes de indexar. Conclusão unânime dos 5: a tabela é uma unidade indivisível de raciocínio — nunca fragmente o que foi concebido como um todo.

P1

Preserve a estrutura — não linearize tabelas em texto. SAT-Graph, SQL ou hipergrafo: a relação linha×coluna é informação, não metadado. Perder estrutura = perder capacidade de raciocínio.

P2

Granularidade múltipla — a mesma tabela pode ser indexada em nível de célula, linha, tabela ou documento. Multi-view não é redundância — é cobertura de perguntas de naturezas distintas.

P3

Simbólico + neural — SQL não substitui embeddings, e embeddings não substituem SQL. A dualidade permite que cada paradigma cubra os pontos cegos do outro: precisão simbólica + generalização semântica.

P4

Qualidade na entrada — SA-RAG prova: metadados de qualidade e normalização antes do indexamento determinam o teto do retrieval. Lixo estruturado ainda é lixo.
