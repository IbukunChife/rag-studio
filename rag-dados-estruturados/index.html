O Problema

Tabelas Não São
Parágrafos

O RAG clássico lineariza tabelas em texto — e perde a estrutura relacional que dá significado aos dados. Raciocínio sobre múltiplas colunas, cálculos de agregação e consultas multi-tabela exigem uma abordagem fundamentalmente diferente.

❌

Perda de Estrutura

Linearizar tabelas em Markdown destrói relações entre linhas e colunas, tornando raciocínio multi-hop inviável.

❌

Sem Visão Global

Chunking fragmenta a tabela — a pergunta "quantos no total?" exige a tabela inteira, não um trecho.

❌

Ruído Conversacional

Em dados de chatbot, o conhecimento útil está enterrado em dezenas de turnos irrelevantes e redundantes.

A solução unificadora: tratar a tabela como uma unidade indivisível de raciocínio, seja via SQL simbólico, seja via grafos semânticos, seja via índices hierárquicos que preservam contexto de linha ↔ coluna.

Metáfora · T-RAG

Biblioteca com
Índice em Três Dimensões

Metáfora do Cotidiano

Imagine uma biblioteca gigante com 57 mil livros. Uma abordagem ingênua lista todos os títulos em ordem alfabética. O T-RAG é como um bibliotecário que organiza os livros de três formas simultâneas: pelo assunto (semântica), pelo tamanho e formato (estrutura) e pela frequência de consulta (heurística). Quando você faz uma pergunta, ele vai direto ao cluster certo nas três dimensões — e então afina a busca com um subgrafo local. Nenhum livro irrelevante é sequer tocado.

Hipergrafos Heterogêneos

Tabelas são nós do grafo; hiperarestas agrupam tabelas similares em três tipos: semântica (xsem), estrutura (xstruct), heurística (xheur).

Busca Grossa → Fina

Fase 1: seleciona o cluster correto para cada dimensão. Fase 2: executa busca de subgrafo local. Fase 3: prompting ciente do grafo para LLM.

T-RAG · Arquitetura

Hipergrafo como
Memória Hierárquica

A linearização de tabelas para sequências de texto aumenta complexidade e perde estrutura. T-RAG abandona esse caminho e constrói um hipergrafo heterogêneo onde cada tabela tem representação composta em três espaços distintos.

  • Linearização canônica: s = [Table] ⊕ [Caption, A] ⊕ Σ[Header, h_k] — preserva título e headers sem perder contexto estrutural.
  • KMeans tridimensional: cada tipo de feature gera um conjunto de hiperarestas, formando hiperedges heterogêneos ε ∈ {sem, struct, heur}.
  • Seleção de nós típicos: para cada cluster C_j, extrai top-k nós representativos — reduz comparações de queries às tabelas protótipo.
  • Graph-aware prompting: injeta priors relacionais estruturados na entrada do LLM, melhorando raciocínio cross-table.
  • Avaliação comparativa: supera table retrieval methods, RAG methods e table-to-graph learning em acurácia, recall e eficiência de tempo.
Metáfora · FT-RAG

A Lupa que Lê
Cada Célula

Metáfora do Cotidiano

Imagine um analista financeiro que precisa encontrar a receita doméstica do Segmento X no terceiro trimestre de 2024 num relatório de 200 páginas. O RAG convencional escaneia o texto em blocos — como folhar rápido. O FT-RAG é como uma lupa especial: ela não lê "a tabela", ela lê cada célula individualmente, sabendo exatamente que "12.11" está na linha Domestic, coluna October, Segment X, Table 2, Company A — com toda a cadeia de contexto preservada em coordenadas precisas. Quando você perguntar "qual foi a receita?", ela vai direto à célula certa sem ambiguidade.

Cell Groups (Unidades Semânticas)

Cada célula vira uma Cell Group com: valor bruto + índices (coordenadas) + tipo (row/col header) + hierarquia (nível de nesting). Contexto completo embutido.

SAT-Graph (Grafo Temporal)

Subject Sub-tree (Ts) + Temporal Sub-tree (Tt) + Attribute Node Set (Va) + Value Leaf Node Set (Vl). DAG composto e direcionado.

FT-RAG · SAT-Graph

O Grafo que Captura
Sujeito, Atributo e Tempo

O SAT-Graph resolve a perda de contexto orthogonal das tabelas 2D: uma linha tem sujeito, uma coluna tem atributo, e os dados têm validade temporal. Esses três eixos são transformados em sub-árvores explícitas.

Subject Sub-tree (Ts)

Inferido via LLM sobre o contexto global da Cell Group. Reconstrói hierarquia de sujeitos — "Laptop" aninhado sob "Hardware". Identidade + linhagem estrutural.

Temporal Sub-tree (Tt)

Derivado via parsing de padrões. Organiza intervalos de tempo cronologicamente — October, November, December como scopes de validade do dado.

Benchmark Multi-Table-RAG-Lib: 9.870 pares QA de alta complexidade — 5.000 tabelas, 308 tabelas grandes, 81 documentos. Contextual flag f=1 exige fusão texto+tabela para 4.935 questões. A outra metade (f=0) exige tabela pura, isolando a capacidade de raciocínio tabular do LLM.

Metáfora · TableRAG

O Detetive que
Interroga o Banco de Dados

Metáfora do Cotidiano

Um detetive ruim faz perguntas vagas e espera que as testemunhas entreguem a resposta. Um detetive expert sabe exatamente qual pergunta fazer: "Me dê os nomes dos suspeitos que estavam na cena entre 21h e 23h, ordenados pela hora de saída." O TableRAG usa SQL como linguagem de interrogação precisa. Em vez de recuperar chunks de tabela que podem estar incompletos, ele gera e executa consultas SQL exatas — tratando a tabela como uma unidade indivisível de raciocínio, não como parágrafos fatiados.

SQL como Linguagem Formal

f_SQL(S^t, q_t) = e_t — execução simbólica precisa sobre dados relacionais. Python introduz overhead; SQL é indivisível e determinístico.

Dois Corpora Paralelos

Fase offline: knowledge base textual (chunks do texto T) + schema database tabular (D). Fase online: raciocínio iterativo híbrido sobre ambos.

TableRAG · Pipeline

4 Passos de
Raciocínio Iterativo

Fase Offline

Construção de banco de dados: parsing de tabelas → schema extraction → MySQL + knowledge base vetorial. Mapeamento f: D̂_{i,j} → S(D_i) preserva vínculo chunk-schema.

Fase Online

Iteração até convergência: decomposição → retrieval → SQL (quando chunk vem de tabela) → resposta intermediária. Fusão compositiva de ambas as fontes.

Benchmarks existentes testam apenas retrieval superficial (resposta direta em célula). O HeteQA exige raciocínio multi-hop que cruza texto e tabelas — 5 operações tabulares compostas por questão.

+17,3%

vs. RAG Genérico

Sobre baselines genéricos de RAG no HeteQA — TableRAG estabelece novo SOTA para QA em documentos heterogêneos texto+tabela.

  • Context-Sensitive Query Decomposition: identifica quando a subquery precisa de tabela (→ SQL) ou texto (→ retrieval vetorial). Não é apenas parsing sintático — requer consciência da fonte de dados.

Metáfora · NeuSym-RAG

O Arquivista que
Fala Duas Línguas

Metáfora do Cotidiano

Num arquivo de pesquisa, há dois sistemas: o índice fuzzy — onde você busca por aproximação ("algo sobre redes neurais em 2023") — e o banco de dados SQL — onde você pergunta exatamente ("papers com mais de 50 citações publicados entre janeiro e março de 2023"). Um arquivista que fala só uma das línguas sempre falha em alguma pergunta. O NeuSym-RAG é o arquivista bilíngue que decide qual língua usar dependendo da pergunta — e às vezes usa as duas em sequência, refinando o resultado.

NeuSym-RAG · Multiview

Multi-View Parsing:
Quatro Lentes no Mesmo PDF

Parsing Semi-estruturado

PyMuPDF para texto multi-granular. MinerU (OCR avançado) para elementos não-textuais: tabelas, figuras, fórmulas. Metadados via API do ArXiv. Tudo → DuckDB relacional.

Encoding Multimodal

BM25 + bge-large-en-v1.5 para texto. clip-vit-base-patch32 para imagens. Colunas "encodable" no DB schema → vectorstore com 4 coleções distintas por modalidade.

NeuSym-RAG · Agente

5 Ações para
Recuperação Iterativa

O agente NeuSym-RAG segue o framework ReAct — Thought → Action → Observation — iterando até ter contexto suficiente. A chave é a combinação simbólica+neural em cada passo.

  • RetrieveFromVectorstore: busca fuzzy semântica com reescrita de query via LLM (Ma et al. 2023). Suporta filtro por table_name, column_name, page_number.
  • RetrieveFromDatabase: executa SQL contra DuckDB. Ideal para queries com filtros exatos, ordenação, contagem e aggregation — o que o vectorstore não sabe fazer.
  • ViewImage: extrai região (bounding_box) de uma página do PDF em base64. Resolve erros de OCR e tabelas com layout visual complexo — o agente "vê" o que não consegue parsear.
  • CalculateExpr: avalia expressão Python aritmética (e.g., 2 + 3 × 4). Reduz alucinação em problemas matemáticos — resultado computado, não gerado.
  • GenerateAnswer: ação terminal. Ativada quando o agente julga que o contexto recuperado é suficiente para responder com fidelidade.
Metáfora · SA-RAG

A Triagem do
Pronto-Socorro

Metáfora do Cotidiano

Um pronto-socorro recebe pacientes em estado caótico: gritos, dores de todo tipo, informações contraditórias, muita gente falando ao mesmo tempo. A triagem não tenta processar tudo — ela converte o caos em uma ficha estruturada: nome, sintoma principal, urgência, histórico relevante. O SA-RAG faz o mesmo com dados conversacionais: cada conversa barulhenta de chatbot é convertida em uma tabela limpa, com metadados de qualidade, eliminando turnos irrelevantes e normalizando o conhecimento útil antes de qualquer busca.

Quality-Aware Metadata

L(M) = Σ R(m_ij, C) + S(M) — função de qualidade que avalia relevância e consistência de cada entrada de metadado gerada pelo LLM.

DPO para Tabelas

L_DPO(θ) minimiza distância entre resposta preferida e rejeitada, guiando a geração de tabelas para consistência semântica e estrutural.

SA-RAG · Pipeline

Tabela como
Filtro de Ruído

Training-Free

Semantic consistency: p_ij = 1_sem(φ_ij, D_i) · 1_struc(φ_ij, M_j). Validação sem fine-tuning — funciona com qualquer LLM como gerador de tabela.

Training-Based (DPO)

Preferência direta otimiza para high-quality tables. Um único processo de treinamento satisfaz simultaneamente os constraints de qualidade, semântica e consistência estrutural.

SA-RAG · Resultados

RAG Estrutural para
Agentes Conversacionais

Em dados conversacionais limpos (Wikipedia), todos os métodos competem bem. A vantagem do SA-RAG aparece em dados ruidosos reais — onde texto-RAG e graph-RAG falham por sensibilidade ao ruído ou dependência de extração de entidades precisa.

Por que Tabelas Vencem Grafos no Ruído?

Graph-RAG exige extração precisa de entidades e relações — em dados ruidosos, erros de extração propagam. Tabelas são mais tolerantes: campos parcialmente preenchidos e atributos ausentes não quebram o retrieval.

Normalização como Chave

Dados conversacionais têm nomes inconsistentes ("app A", "aplicativo A", "AppA"). SA-RAG normaliza esses atributos no schema gerado, garantindo que queries diferentes encontrem o mesmo fato.

Avaliação em dois datasets ruidosos reais: conversações de suporte técnico e chats de e-commerce — contextos onde as informações úteis estão misturadas a saudações, clarificações e repetições. SA-RAG supera text-RAG, graph-RAG e simple TableRAG em ambos.

Síntese

Princípios do RAG
Estruturalmente Consciente

  • Princípio 1 — Preserve a Estrutura: não linearize tabelas em texto. Seja via SAT-Graph (FT-RAG), schema SQL (TableRAG) ou hipergrafos (T-RAG), a estrutura relacional é informação, não metadado.
  • Princípio 2 — Granularidade Múltipla: a mesma tabela pode ser indexada em nível de célula, linha, tabela ou documento. Perguntas diferentes precisam de granularidades diferentes. Multi-view não é redundância — é cobertura.
  • Princípio 3 — Simbólico + Neural: SQL não substitui embeddings, e embeddings não substituem SQL. A dualidade permite que cada paradigma cubra os pontos cegos do outro — precisão simbólica + generalização semântica.
  • Princípio 4 — Qualidade na Entrada: lixo estruturado ainda é lixo. SA-RAG mostra que metadados de qualidade, normalização e validação antes do indexamento determinam o teto do retrieval.
  • Princípio 5 — Raciocínio como Unidade: uma subquery tabular é indivisível. Não recupere pedaços de tabela — recupere a tabela inteira e execute a operação como um bloco atômico.
