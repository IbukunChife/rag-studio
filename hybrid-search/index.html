O Problema

Por Que uma Busca Só Não Basta para RAG?

Todo sistema RAG precisa recuperar documentos relevantes antes de gerar uma resposta. O problema é que nenhum método de recuperação isolado funciona bem em todos os cenários.

🔵

BM25 (Léxico)

Excelente em termos exatos. Cego para paráfrases e sinônimos — se a palavra não bate, não recupera.

🟣

Dense (Semântico)

Captura contexto e significado. Dilui termos técnicos, siglas e entidades específicas de domínio.

⚖️

Alpha Fixo

Combinar os dois com peso fixo (α = 0,5) é uma solução medíocre — nem léxica nem semântica se beneficia plenamente.

A tese central desta categoria: busca híbrida é a base certa — mas como calibrar o peso, lidar com dados ao vivo, verificar a resposta e escolher entre 10 estratégias? São exatamente essas 4 perguntas que os artigos desta categoria respondem.

Fundamentos

O Paradoxo da Busca Única

BM25 e Dense enxergam o mundo de formas opostas — e é exatamente essa diferença que torna a fusão deles poderosa.

BM25 — "redes neurais" ❌

A query usa o termo português. Os documentos usam "neural networks". BM25 não recupera — as palavras não batem.

Dense — "EBITDA 2024" ⚠️

O embedding dilui identificadores precisos. "EBITDA Q3 2024" e "lucro antes de impostos" ficam parecidos — mas o dado exato se perde.

Fundamentos

A Fusão que Supera os Dois

Combinar BM25 e Dense via Reciprocal Rank Fusion (RRF) produz um ranking superior ao de qualquer método isolado — validado em todos os 4 artigos desta categoria.

🔵

BM25

Correspondência léxica — termos exatos, siglas, entidades nomeadas

🟠

RRF — Fusão

RRF(d) = Σ 1/(k+rᵢ) — combina rankings sem precisar normalizar scores

🟣

Dense

Embeddings semânticos — paráfrases, sinônimos, contexto implícito

Artigo 01 · DAT
Dynamic Alpha Tuning · Benchmarks SQuAD / DRCD

Quando o Alpha Certo Muda Tudo

A busca híbrida usa a fórmula R(q,d) = α·Ŝdense + (1-α)·ŜBM25. O DAT pergunta: por que α seria o mesmo para toda query?

Um α fixo (tipicamente 0,5) trata queries semânticas e léxicas do mesmo jeito — desperdiçando precisão nos dois extremos. Uma query sobre sinônimos precisa de α alto; uma query com siglas técnicas precisa de α baixo.

Usar um LLM para pontuar o Top-1 de cada método (BM25 e Dense) numa escala de 0 a 5, e calcular automaticamente o α ideal para aquela query específica — sem treino adicional.

+7,5%
de ganho em queries híbrido-sensíveis — as mais difíceis para α fixo. Nos próximos slides: a fórmula exata e os resultados completos.
Artigo 01 · DAT — Mecanismo

α(q) = Sv / (Sv + SBM25)

O LLM avalia o Top-1 de cada método isoladamente e os scores derivam o α automaticamente — uma query de cada vez, em tempo de inferência.

α(q) = Sv / (Sv + SBM25)

Passo 1 — Recuperar

BM25 e Dense recuperam seus Top-K independentemente. O Top-1 de cada um é separado para avaliação.

Passo 2 — Pontuar

LLM avalia cada Top-1 numa escala 0→5: quão relevante é este documento para esta query?

Passo 3 — Calcular α

S_v e S_b normalizam para α ∈ [0,1]. Se Dense vence (S_v=4, S_b=1) → α=0,80. Vice-versa → α=0,20.

QUERY SEMÂNTICA
S_v=4, S_b=1 → α = 0,80
Dense domina o ranking final
QUERY LÉXICA
S_v=1, S_b=4 → α = 0,20
BM25 domina o ranking final
CUSTO ADICIONAL
1 chamada LLM por query
Zero custo de treino ou fine-tuning

Artigo 02 · LiveRAG
LiveRAG Challenge 2025 · 4º Fidelidade, 11º Correção

O Desafio dos Dados que Não Param

E quando os documentos são atualizados continuamente? O LiveRAG Challenge 2025 testou sistemas RAG sobre um corpus em streaming — e a resposta surpreendeu.

Em dados ao vivo (streaming), os documentos mudam antes mesmo do sistema terminar de indexá-los. Estratégias elaboradas como doc2query e DSPy falham ou criam novos problemas (overconfidence, latência absurda).

O fator que mais impactou a performance foi o mais simples: alinhamento vocabular entre a query e os documentos. Antes de otimizar arquitetura, otimize vocabulário.

+52%
MAP com RankLLaMA re-ranker (0,523 → 0,797) — mas com 84 segundos por query. O artigo expõe o trade-off brutal entre qualidade e latência em re-ranking neural.
Artigo 02 · LiveRAG — Contexto

RAG sobre um Rio de Dados

Pipeline Completo

O que Falhou

A análise pós-competição revelou: o alinhamento entre vocabulário da query e vocabulário dos documentos dominou todos os fatores técnicos testados.

Artigo 03 · Evidence-Grounded RAG
Amazon Bedrock + Cohere + Claude Sonnet 4.6 + Amazon Nova Pro · Domínio Biomédico

Cada Afirmação Precisa de Prova

RAG gera respostas fluentes — mas fluência não é correção. Como saber se cada frase da resposta está realmente sustentada pelos documentos recuperados?

Avaliar a resposta como um bloco único esconde afirmações sem sustentação. Um LLM pode gerar 5 afirmações corretas e 1 alucinada — e a nota da resposta seria "boa".

Decompor cada resposta em claims individuais e avaliar cada um contra o contexto recuperado, usando um modelo diferente do gerador como juiz — evita viés de autoavaliação.

200/200
claims avaliados — todos sustentados por evidência recuperada. Stack: S3 → Bedrock Knowledge Bases → Titan V2 → OpenSearch → Cohere Rerank → Claude Sonnet 4.6 → Nova Pro (juiz)

Artigo 04 · T²-RAGBench

O Maior Teste de RAG em Documentos Financeiros

Dentre tantas estratégias de recuperação disponíveis, qual realmente funciona quando os documentos misturam texto e tabelas em relatórios financeiros reais?

Não existia benchmark sistemático para RAG em documentos financeiros texto+tabelas. Pesquisadores escolhiam estratégias sem evidência sólida de qual funcionava melhor nesse domínio específico.

T²-RAGBench: 23.088 queries sobre 7.318 documentos SEC (avg 920 tokens). 10 estratégias avaliadas com métricas R@5, MRR@3, nDCG@10. Primeiro resultado sistemático no setor.

R@5 = 0,816
Híbrido RRF + Cohere Rerank vence com margem expressiva (+17pp sobre o 3º lugar). Surpresa: BM25 supera Dense em todos os subsets financeiros. HyDE fica em último.

Artigo 04 · Análise de Falhas

Onde o Melhor Método Ainda Erra

31,1% das queries (7.188/23.088) não encontraram o documento certo no Top-5. A causa dominante não é o método de fusão — é a estrutura dos dados.

73% — Estrutura de Tabela

A resposta está numa célula de tabela em Markdown. O embedding não captura a posição estrutural — a coluna certa e o row certo dentro do layout.

20% — Raciocínio Numérico

A query pede uma conta ("variação anual?"). O dado bruto existe no documento, mas a query requer dois números + operação — semântica pura não localiza isso.

Síntese · 4 Artigos

A Pirâmide da Recuperação Híbrida

Os 4 artigos constroem uma pirâmide coerente: cada camada adiciona precisão sobre a anterior, com custo progressivo e ganho comprovado nos estudos.

Regra prática: comece com Híbrido RRF como baseline. Adicione re-ranker se qualidade supera latência (Art. 04 confirma +17pp). Aplique alpha dinâmico se queries variam muito em tipo (Art. 01 confirma +7,5%). Avalie com claim-level, não answer-level (Art. 03). Não esqueça vocabulário (Art. 02).

Guia Prático · Decisão

Qual Estratégia para o Seu Caso?

A escolha depende do domínio, tipo de query e tolerância à latência. Este mapa consolida as recomendações diretas dos 4 artigos.

Documentos Financeiros / Técnicos
  • → Baseline mínimo: BM25 (supera Dense em finanças — Art. 04)
  • → Melhor opção: Híbrido RRF + Cohere Rerank
  • → Evitar: HyDE (alucinações numéricas corrompem o embedding)
Dados ao Vivo / Streaming
  • → Baseline: Híbrido BM25 + E5 — Art. 02
  • → Foco primeiro: alinhe vocabulário antes de otimizar arquitetura
  • → Cuidado: re-ranker LLM tem 84s/query — avalie o SLA
Alta Confiabilidade / Compliance
  • → Pipeline: Híbrido + Cohere + LLM gerador — Art. 03
  • → Avaliação: juiz independente, claim a claim
  • → Meta: 100% grounding accuracy por afirmação
Queries Heterogêneas
  • → Baseline: Híbrido RRF
  • → Avançado: DAT — alpha dinâmico por query — Art. 01
  • → Trade-off: 1 chamada LLM extra por query para calibrar α
Conclusão · Síntese Final

Quatro Descobertas que Redefinem o RAG

Cada artigo respondeu uma pergunta diferente — juntos eles formam um manual atualizado de como construir recuperação híbrida de qualidade.

  • 1
    Alpha fixo desperdiça potencial (DAT · Art. 01)

    Um α único para todas as queries deixa +7,5% na mesa. O LLM como árbitro por query é a solução mais eficiente sem custo de treino.

  • 2
    Vocabulário importa mais que arquitetura (LiveRAG · Art. 02)

    Em dados ao vivo, a lacuna vocabular explica 26% de queda em performance — sem mudar nenhum componente do pipeline. Corrija isso primeiro.

  • 3
    Grounding precisa ser claim a claim (Evidence-RAG · Art. 03)

    Avaliar a resposta inteira esconde alucinações. Decompor em afirmações individuais com juiz independente alcançou 100% de grounding em domínio biomédico.

  • 4
    Reranker é o maior ganho unitário disponível (T²-RAGBench · Art. 04)

    Em 10 estratégias avaliadas, adicionar cross-encoder reranker ao híbrido foi o maior salto isolado: +17pp MRR@3. É o componente com melhor custo-benefício hoje.
