Receita Fixa vs. Chef Adaptativo

Um RAG com configuração estática é como seguir sempre a mesma receita: "retrieval de 10 chunks, sumarize cada um separado". Para uma pergunta simples ("Em que ano nasceu Einstein?") é como fazer bolo de casamento para uma lanche rápida — caro, lento e desnecessário.

O METIS é o chef experiente: adapta a receita para cada pedido. Pergunta simples → 1 chunk, resposta direta. Pergunta complexa ("Compare os custos operacionais da NVIDIA nos três últimos trimestres") → 5 chunks, leitura conjunta, sumarização intermediária. Mesma qualidade, tempo ideal para cada prato.

Config Estática (Receita Fixa)
Mesmos parâmetros para todas as queries. Simples ficam lentas (excesso de contexto), complexas ficam pobres (poucos chunks). Latência desnecessária ou qualidade degradada.
Config Adaptativa METIS
LLM profiler estima complexidade → seleciona configuração do espaço podado. 12-15% mais qualidade, 1.64-2.54× menos latência vs. melhor config estática.
Ray et al. · U Chicago + Princeton · SOSP 2025

Por Que a Config Estática Falha

Queries diferem em complexidade — e a configuração ótima varia drasticamente.

"Choosing the per-query configuration allows up to 3× delay saving compared to the static configuration that is the closest in quality. Every single static configuration that achieves comparable delay has at least a 10% quality drop."
— Ray et al., METIS: Fast Quality-Aware RAG Systems with Configuration Adaptation, SOSP 2025

Arquitetura METIS

3 Knobs + Query Profiling

METIS controla 3 parâmetros-chave que afetam simultaneamente qualidade e latência.

🔢
Knob 1: num_chunks
Quantos chunks recuperar. Simples: 1-2. Complexas: 5-15. Mais chunks = mais contexto mas mais GPU e delay.
🔀
Knob 2: synthesis_method
stuff: concatena tudo. map_rerank: lê separado, pega melhor. map_reduce: resume cada + sintetiza. Custo: stuff < rerank < reduce.
📏
Knob 3: intermediate_length
Tamanho do resumo intermediário no map_reduce (1-100 palavras). Simples: 10-20. Complexas: 70-100.
Query Profiler LLM: Analisa a query (quantas peças de informação são necessárias?) → gera query profile → poda espaço de configuração para conjunto menor mas promissor → scheduler seleciona config otimizando latência dentro do espaço podado.

Metáfora · Multi-Objetivo HPO
🚗
Compra de Carro pela Fronteira de Pareto

Ao comprar um carro, você quer o mais rápido, o mais econômico, o mais seguro E o mais barato. Mas nenhum carro domina todos os outros em todos os critérios ao mesmo tempo.

A Fronteira de Pareto é a lista dos "melhores compromissos possíveis": o carro mais rápido dado um orçamento, ou o mais econômico dada uma velocidade mínima. Nenhum carro fora da fronteira é ótimo — há sempre um dentro que é melhor em tudo.

O Multi-Obj HPO aplica esse raciocínio ao RAG: encontra configurações onde melhorar custo não piora qualidade, e melhorar latência não compromete safety.

Barker et al. · Trustwise AI + NYU · ICLR 2025 Workshop

Nuances Críticas · Multi-Obj HPO

Config Ótima Não Generaliza

Task Dependence

A configuração Pareto-ótima para FinancialQA (modelo maior, chunk grande, reranking) pode ser sub-ótima para MedicalQA (termos técnicos precisam de embedding especializado, chunk menor).

Implicação prática: calibre HPO por domínio, não uma vez para tudo.

Objective Dependence

Safety e Alignment seguem tendências opostas para algumas configurações: o modelo mais "safety-aligned" gera respostas mais curtas e recusa mais, reduzindo helpfulness.

Implicação prática: defina pesos explícitos entre objetivos antes da otimização.

"Optimal configurations may not generalize across tasks and objectives — highlighting nuances such as how optimal configurations may not generalize across tasks and objectives."
— Barker et al., Faster, Cheaper, Better, ICLR 2025 Workshop
Metáfora · HPO Greedy
🕵️
O Detetive Greedy — Evidência Mais Valiosa Primeiro

Um bom detetive não examina todas as pistas em ordem de aparição — começa pela mais reveladora. Se as impressões digitais já identificam o culpado, perder tempo medindo as pegadas é desperdício.

O HPO Greedy para RAG funciona igual: em vez de otimizar parâmetros na ordem do pipeline (chunking → embedding → retrieval → geração), começa pelo modelo generativo e embedding — que têm o maior impacto na qualidade. Depois ajusta chunk size, overlap e top-k.

Resultado: convergência 2-3× mais rápida com desempenho superior ao TPE e random search.

Orbach et al. · IBM Research · AAAI 2026

Descobertas · IBM HPO

Greedy-M Vence: Modelo Primeiro!

Greedy-M (Modelo Primeiro)

Otimiza: Generative Model → Embedding Model → Chunk Size → Chunk Overlap → Top-K.

Lógica: modelo generativo domina qualidade. Fixando-o primeiro, os demais parâmetros convergem mais rápido.

✓ Melhor desempenho médio em todos os datasets
Greedy-R (Retrieval Primeiro)

Otimiza: Embedding → Chunk Size → Chunk Overlap → Generative Model → Top-K.

Segue a ordem natural do pipeline RAG. Intuitivo, mas sub-ótimo — retrieval sem saber o modelo final é míope.

✗ Converge mais lento, desempenho inferior
Random Search surpresa: significativamente melhor que grid search nas primeiras iterações — a randomização evita padrões que travam em mínimos locais. HPO com apenas 10-20 iterações já alcança 90%+ do ótimo grid.
Wang et al. · Fudan University · Jul 2024

Melhores Práticas por Módulo RAG

Estudo sistemático de combinações por componente — de Query Classification a Fine-tuning.

  1. 1
    Query Classification
    BERT-based classifier (0.95 F1) decide se retrieval é necessário. Evita overhead para queries que o LLM já sabe responder.
  2. 2
    Chunking Best Practice
    Small2Big (chunk pequeno para retrieval, chunk grande para contexto) + Sliding Windows para documentos longos.
  3. 3
    Embedding
    bge-large-en-v1.5 (BAAI/bge) melhor geral. LLM-Embedder para casos domain-specific. Treinar contrastivamente quando possível.
  4. 4
    Retrieval
    HyDE (Hypothetical Document Embeddings) + Hybrid Search melhor combinação. Query Rewriting melhora casos ambíguos.
  1. 5
    Reranking
    monoT5 melhor equilíbrio qualidade × velocidade. TILDE (precomputed) para latência extrema. DLM-based para máxima qualidade.
  2. 6
    Repacking (Ordering)
    Estratégia "Sides" vence: docs relevantes no início E no fim — combate o "lost-in-the-middle" do LLM.
  3. 7
    Summarization
    Recomp (extractive) bom equilíbrio qualidade/custo. SelectiveContext para contextos muito longos.
  4. 8
    Multimodal — "Retrieval as Generation"
    Para conteúdo visual: gerar texto descritivo a partir de imagem → indexar texto → recuperar normal. Sem modelo multimodal necessário.

Insights · Best Practices

4 Insights que Surpreendem

Nem toda query precisa de RAG
Query Classification (BERT 0.95 F1) filtra queries que o LLM já sabe responder. Retrieval desnecessário adiciona latência sem ganho de qualidade.
Lost-in-the-middle é real e solucionável
LLMs ignoram docs no meio do contexto. Repacking "Sides" (relevante no início E fim) combate o problema — sem custo adicional de compute.
HyDE vale o custo extra
Gerar documento hipotético antes de buscar melhora recall significativamente. O overhead de 1 chamada LLM extra é compensado pela melhora de retrieval.
Fine-tuning conjunto retriever+gerador
Fine-tuning holístico (retriever e gerador juntos) supera ajuste de módulos isolados — mas aumenta complexidade. Use somente quando qualidade é crítica.

Síntese do Módulo 13

5 Lições sobre Otimização de Pipeline

  1. 1
    Config por query > config estática
    METIS: adaptar 3 knobs por query alcança 15% mais qualidade E 2.54× menos latência vs. melhor config fixa. Não existe um-tamanho-para-todos.
  2. 2
    HPO multi-objetivo tem compromissos inevitáveis
    Não existe a melhor config: existe a fronteira de Pareto. Safety vs. alignment podem conflitar. Defina pesos explícitos entre objetivos antes de otimizar.
  3. 3
    Otimize o modelo primeiro
    IBM: Greedy-M (modelo generativo + embedding primeiro) converge mais rápido e atinge melhor resultado. 10-20 iterações HPO são suficientes.
  4. 4
    Pipeline tem quick wins sem custo
    Best Practices: Query Classification evita retrieval desnecessário; Sides repacking combate lost-in-the-middle; HyDE melhora recall — ganhos de engenharia sem retraining.
  5. 5
    Config ótima não generaliza entre domínios
    FinancialQA ≠ MedicalQA ≠ enterprise docs. Execute HPO por domínio. Random search nas primeiras iterações é surpreendentemente eficaz.
Guia Prático de Otimização

Por Onde Começar a Otimizar?

  1. 1
    Quick wins de engenharia (custo: zero)
    Implementar Query Classification, Sides repacking, HyDE para retrieval. Ganhos imediatos sem custo de treinamento.
  2. 2
    HPO greedy (10-20 iterações)
    Testar 3 modelos generativos × 3 embeddings primeiro (Greedy-M). Depois chunk size e top-k. Random search nas primeiras 5 iterações.
  3. 3
    Config adaptativa por query (se latência é crítica)
    Implementar profiler de complexidade de query. Criar espaço de configurações por tier (simples/médio/complexo). Monitorar tradeoffs quality × latency.
  4. 4
    MOO Pareto (se múltiplos stakeholders)
    Definir objetivos com pesos. BO com qLogNEHVI. Escolher ponto da fronteira de Pareto alinhado com prioridades do negócio.
