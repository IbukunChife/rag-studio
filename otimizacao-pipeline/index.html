Receita Fixa vs. Chef Adaptativo

Um RAG com configuração estática é como seguir sempre a mesma receita: "retrieval de 10 chunks, sumarize com map_reduce". Para uma pergunta simples — "Em que ano nasceu Einstein?" — é como fazer bolo de casamento para um lanche rápido: caro, lento e desnecessário.

O METIS é o chef experiente: adapta a receita para cada pedido. Pergunta simples → 1 chunk, map_rerank, resposta em 0,3s. Pergunta complexa → 10 chunks, map_reduce, sumarização intermediária. Mesma cozinha, cada prato preparado na medida certa.

Config Estática (Receita Fixa)

Mesmos parâmetros para todas as queries. Simples ficam lentas (excesso de contexto), complexas ficam rasas (poucos chunks). Latência desnecessária ou qualidade degradada.

Config Adaptativa METIS

LLM profiler estima complexidade → seleciona config do espaço podado → scheduler otimiza latência. 15% mais qualidade com 2,54× menos latência vs. melhor config estática.

METIS · Ray et al. · U Chicago + Princeton · SOSP 2025

Por Que a Config Estática Sempre Perde

Cada query tem uma configuração ótima diferente — forçar um único template é um desperdício estrutural

"Choosing the per-query configuration allows up to 3× delay saving compared to the static configuration that is closest in quality. Every single static configuration that achieves comparable delay has at least a 10% quality drop."
— Ray et al., METIS: Fast Quality-Aware RAG Systems with Configuration Adaptation, SOSP 2025

Q1 · Simples

"Em que condado nasceu William Blair?" → 1 chunk, map_rerank. Mais chunks = 2× delay SEM melhora de qualidade.

Q2 · Média

"Alison, Diane e Rachel são do mesmo país?" → 3-5 chunks, leitura conjunta (+35% qualidade vs. map_rerank).

Q3 · Complexa

"Voyager 1 detectou tempestades em Netuno quando?" → 10+ chunks, map_reduce. Sem ele: qualidade cai 40%.

Animação Conceitual · O Profiler de Complexidade METIS

Query Entra — Config Nasce na Hora

O LLM profiler classifica a complexidade da query e seleciona automaticamente os 3 knobs ótimos — sem intervenção humana

Knob 1: num_chunks

Simples: 1-2 · Médio: 3-5 · Complexo: 8-15

Knob 2: synthesis_method

stuff → map_rerank → map_reduce (custo crescente)

Knob 3: summary_length

map_reduce: 10-20 palavras (simples) · 70-100 (complexo)

Comprar Carro pela Fronteira de Pareto

Ao comprar um carro, você quer o mais rápido, o mais econômico, o mais seguro e o mais barato. Mas nenhum carro domina todos os outros em todos os critérios ao mesmo tempo. A Fronteira de Pareto lista os melhores compromissos possíveis: o mais rápido dado um orçamento, o mais econômico dada uma velocidade mínima. Nenhum carro fora da fronteira é uma escolha inteligente — há sempre um dentro que é melhor em tudo.

O Multi-Obj HPO aplica isso ao RAG: encontra configs onde melhorar custo não piora qualidade, e melhorar safety não compromete alignment. Você escolhe onde quer estar na fronteira — o algoritmo a mapeia toda.

Custo

tokens × preço LLM por query

Latência

tempo end-to-end por resposta

Safety

risco de alucinação ou conteúdo indevido

Alignment

utilidade / helpfulness da resposta

Bayesian Opt: Fronteira de Pareto em RAG

qLogNEHVI (hypervolume indicator) — robusto a avaliações ruidosas · 15+ hiperparâmetros otimizados

Task Dependence: config Pareto-ótima para FinancialQA ≠ MedicalQA — calibre HPO por domínio. Objective Dependence: Safety vs. Alignment frequentemente conflitam — modelo mais "safe" recusa mais, reduzindo helpfulness. Defina pesos antes de otimizar.

O Detetive Greedy — Evidência Mais Valiosa Primeiro

Um bom detetive não examina todas as pistas na ordem em que aparecem — começa pela mais reveladora. Se as impressões digitais já identificam o culpado, medir as pegadas é desperdício puro.

O HPO Greedy-M para RAG funciona igual: em vez de otimizar na ordem do pipeline (chunking → embedding → retrieval → geração), começa pelo modelo generativo e embedding — que têm o maior impacto. O IBM provou isso com 162 configurações: Greedy-M alcança 63% de Answer Correctness — superando o próprio Grid Search com apenas 10-20 iterações.

Greedy-M Vence: Modelo Primeiro!

5 datasets · AIArxiv · BioASQ · MiniWiki · ClapNQ · WatsonxQA — 3 métricas: Context Correctness, Faithfulness, Answer Correctness

Greedy-M (Modelo Primeiro) — VENCEDOR

Ordem: Gen. Model → Embedding → Chunk Size → Overlap → Top-K. Lógica: modelo generativo domina qualidade — fixá-lo primeiro acelera convergência dos demais parâmetros.

Greedy-R (Retrieval Primeiro) — Sub-ótimo

Ordem natural do pipeline RAG. Intuitivo mas míope — otimizar embedding sem saber o modelo final é como escolher acompanhamentos antes de saber o prato principal.

Melhor geral. LLM-Embedder para domain-specific. Treine contrastivamente.

4
Retrieval: HyDE + Hybrid Search

Melhor combinação. HyDE gera doc hipotético antes de buscar — melhora recall.

5

Melhor equilíbrio qualidade × velocidade. TILDE para latência extrema.

6

Docs relevantes no início E no fim — combate o "lost-in-the-middle" do LLM.

7

Extractive — bom equilíbrio qualidade/custo. SelectiveContext para contextos longos.

8

Gera texto descritivo da imagem → indexa texto → recupera normal. Sem modelo multimodal.

Tensão Conceitual · O Debate

Adaptar por Query ou Otimizar Globalmente?

🍳 METIS (Per-Query Adaptation)

Tese: cada query merece sua própria receita. O profiler LLM adiciona uma chamada extra, mas os ganhos de latência e qualidade compensam: 2,54× menos latência, 15% mais F1 vs. qualquer config estática. Custo do profiler é amortizado pela eficiência per-query.

F1 0.65 · Delay 0.5s · overhead do profiler: ~50ms
🕵️ IBM Greedy-M (Global HPO)

Tese: uma boa configuração global, encontrada de forma eficiente, é suficiente para a maioria dos casos. Greedy-M com 10-20 iterações alcança 63% de Answer Correctness — sem overhead de profiling por query. Mais previsível em produção, sem dependência de LLM intermediário.

63% AC · supera grid search · 10-20 iterações

Síntese: Não são opostos — são layers. Use Greedy-M primeiro para encontrar a melhor config base (10-20 iterações, offline). Depois, se latência ou qualidade ainda variam muito entre queries, adicione o profiler METIS como camada de adaptação dinâmica em cima. O custo do profiler (~50ms) compensa quando a variação de complexidade é alta. Se o corpus é homogêneo, Greedy-M sozinho é suficiente.

Módulo 13 · Princípios Destilados

Os 3 Especialistas em Calibração Finalmente de Acordo

Lembram dos 3 especialistas? O Chef Adaptativo (METIS) provou que receita por query é 2,54× mais eficiente — não existe almoço bom com o mesmo setup para jantar. A Fronteira de Pareto (HPO Multi-Obj) mapeou os compromissos inevitáveis entre custo, latência, safety e alignment — você escolhe onde quer estar, ela encontra o ponto. O Detetive Greedy (IBM) descobriu que inspecionar o modelo primeiro acelera a convergência e supera o próprio Grid Search com apenas 10-20 investigações. Conclusão unânime: pipeline ideal = Greedy-M para base (offline) + fronteira de Pareto para calibrar objetivos + METIS para adaptar em runtime.

P1

Config por query supera config estática — METIS prova: qualquer config estática que iguala latência de METIS tem ≥10% queda de qualidade. O overhead do profiler (~50ms) compensa com alta variação de complexidade.

P2

Não existe a melhor configuração — existe a fronteira de Pareto — HPO Multi-Obj: Safety vs. Alignment conflitam, FinancialQA ≠ MedicalQA. Defina pesos por domínio antes de otimizar.

P3

Otimize o modelo generativo primeiro — IBM Greedy-M bate Grid Search com 10-20 iterações. Random search supera TPE nas primeiras 5. Ordem importa mais que algoritmo de busca.

P4

Quick wins de engenharia são gratuitos e imediatos — Query Classification + HyDE + Sides repacking não exigem retraining. Implemente primeiro antes de qualquer HPO.
