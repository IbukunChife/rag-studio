Mapa de Metrô vs. GPS

O RAG vetorial é um GPS: você faz uma pergunta, ele encontra o ponto mais próximo e entrega a rota mais direta. Rápido, eficiente, preciso para destinos simples.

O GraphRAG é o mapa de metrô da cidade: revela conexões que o GPS ignora — "para ir de A a D, transborde em B e passe por C". Mais lento, mais caro, mas indispensável quando o destino exige entender a rede completa de relações.

A questão não é qual é melhor. É saber qual usar para cada viagem.

RAG Vetorial — GPS
Rota direta, busca semântica, baixa latência. Excelente para recuperação de fatos isolados e dados temporais.
GraphRAG — Mapa de Metrô
Navega conexões entre entidades. Indispensável para raciocínio multi-hop e síntese hierárquica de contexto.
Xiang et al. · ICLR 2026

O Paradoxo GraphRAG

GraphRAG frequentemente perde para o RAG vetorial em tarefas do mundo real — e o campo ignorou isso.

"GraphRAG achieves 13.4% lower accuracy on Natural Question compared to vanilla RAG, with particularly poor performance on time-sensitive queries — 16.6% accuracy drop."
— Xiang et al., When to Use Graphs in RAG, ICLR 2026
❌ Onde GraphRAG Falha
• −13.4% em NaturalQuestion vs. RAG vetorial
• −16.6% em queries temporais / dinâmicas
• 2.3× mais latência média que RAG vetorial
• Corpora sem hierarquias explícitas entre entidades
✓ Onde GraphRAG Vence
• +4.5% em raciocínio multi-hop (HotpotQA)
• Síntese de contexto com dependências hierárquicas
• Geração criativa e sumarização contextual
• Knowledge-intensive domains (CS, Direito, Medicina)

3 Cenários Onde o Grafo Domina

Raciocínio Multi-Hop
Perguntas que encadeiam múltiplos fatos via entidades conectadas. +4.5% no HotpotQA vs. RAG vetorial.
🏛️
Hierarquia de Conceitos
Corpus com dependências hierárquicas explícitas: domínio → subdomínio → entidade. GraphRAG navega essa estrutura nativamente.

Sumarização temática e geração criativa onde relações implícitas e contexto disperso importam mais que fatos isolados.
Regra de ouro (ICLR 2026): Se a resposta está em um único parágrafo, use RAG vetorial. Se a resposta requer sintetizar informação dispersa com dependências lógicas entre entidades, considere GraphRAG — mas avalie latência e custo.

Robô Mecânico vs. Arqueólogo Especializado

Construir um Grafo de Conhecimento com LLM é como contratar um arqueólogo especializado: o resultado é rico e preciso, mas cada análise custa caro e leva tempo. Para 1 milhão de documentos, o orçamento explode.

O Dependency Parsing — análise sintática automática da gramática da frase — é o robô mecânico da escavação: extrai triplas ⟨sujeito, verbo, objeto⟩ sem LLM, em escala industrial. E alcança 94% da qualidade do arqueólogo a uma fração mínima do custo.

Para enterprise com corpora gigantes, o robô mecânico é a única opção viável.

LLM-based Extraction
Alta qualidade semântica. Custo ~50-100× maior. Latência proibitiva. Impraticável para corpora enterprise de grande porte.
Dependency Parsing (SpaCy)
94% da qualidade LLM. Custo ~3-5×. Paralelo, escalável. Passive voice handling + phrasal merging incluídos.
Min et al. · SAP Palo Alto · Dez 2025

GraphRAG Enterprise: Sem LLM na Construção

Pipeline de dupla extração para deploy em produção — foco em legado ABAP/SAP S/4HANA.

  1. 1
    Pré-processamento (Docling + SpaCy)
    PDF/HTML/XLSX → parser Docling → chunking hierárquico (2048 chars, 200 overlap) → segmentação de frases → filtro de frases sem verbo para reduzir ruído.
  2. 2
    Extração de Triplas por Dependency Parsing
    SpaCy extrai ⟨sujeito, relação, objeto⟩ via árvore de dependência gramatical. Passive voice handling normaliza construções passivas → ativas. Phrasal merging une entidades multi-token.
  3. 3
    Fusão e Normalização em Graph DB
    Grafo unificado com embeddings separados para entidades, chunks e relações — três vetores por granularidade.
  4. 4
    Hybrid Retrieval via RRF
    Reciprocal Rank Fusion funde similaridade vetorial + traversal de grafo. Multi-granular: entidades, chunks e relações recuperados e fundidos por score de rank.

Recuperação Híbrida

RRF: Vetor + Grafo em 3 Granularidades

A chave é manter embeddings separados por tipo — entidade, chunk e relação — para buscas multi-granulares.

🔷
Entidades
Embedding de cada nó do grafo. Matching fuzzy via similaridade coseno no espaço de entidades.
📄
Chunks
Embedding dos fragmentos de texto. Recuperação semântica densa — camada RAG vetorial tradicional.
🔗
Relações
Embedding das triplas ⟨subj, rel, obj⟩. Permite busca por tipo de relação entre entidades do grafo.
Reciprocal Rank Fusion (k=60):   score(d) = Σ 1/(k + ranki(d))
Funde rankings de múltiplas listas sem normalização de score — robusto a distribuições diferentes.

Taxonomia de Biblioteca — Do Prateleiro ao Livro

Imagine uma biblioteca com etiquetas hierárquicas pré-definidas: "Ciência da Computação" → "Banco de Dados" → "Indexação" → obra específica. Cada documento carrega essas etiquetas como herança.

O GraphRAG tradicional reconstrói toda a biblioteca a cada novo livro adicionado — caro e lento. O TagRAG pré-organiza as prateleiras com domain tag chains, permitindo inserção incremental: novo livro, nova etiqueta, sem demolir e reconstruir a taxonomia.

A busca navega as etiquetas como um índice de assuntos. Resultado: 14.6× mais rápido na construção, 1.9× mais rápido na busca.

Tao et al. · ECNU · Mai 2026

TagRAG: Hierarquia de Etiquetas de Domínio

Tag Knowledge Graph Construction + Tag-Guided RAG em dois estágios complementares.

① Construção do Tag KG
  1. 1
    Object Tag Extraction: LLM extrai palavras-chave de domínio dos chunks → object tags (Vo, Eo) com descrições
  2. 2
    Domain Tag Chains: Associa tags ao root domain (v̂) → cadeias hierárquicas organizadas em DAG acíclico
  3. 3
    Knowledge Fusion: s = LLM(Chain(vd), Nei(vd)) — sumarização funde informação nas domain tags
② Tag-Guided RAG
  1. 1
    Domain Tag Retrieval: Query → recupera domain tags mais relevantes via similaridade
  2. 2
    Hierarchical Chain Integration: Cadeias hierárquicas coletadas → contexto global sintetizado
  3. 3
    Fused Generation: LLM gera resposta com visão domain-centric — local e global integrados

ROGRAG: 4 Frameworks em 1 Pipeline Robusto

Funde os melhores de DB-GPT, LightRAG, KAG e HuixiangDou com ablação rigorosa de cada componente.

4 Frameworks Fundidos
DB-GPT LightRAG KAG HuixiangDou
Escalabilidade (DB-GPT) + implementação leve (LightRAG) + raciocínio (KAG) + robustez a refusals (HuixiangDou).
2 Estratégias de Retrieval
Logic Form Retrieval: LLM decompõe query em operadores estruturados (filtrar, agregar, ordenar) → traversal estruturado do grafo.

Dual-Level Fuzzy Match: Low-level (entidades) + High-level (relações) → fuzzy matching simultâneo no grafo.
Retrieval Verifier: LLM verifica se o contexto recuperado é suficiente. Se não, degrada automaticamente para Dual-Level Fuzzy — garantindo cobertura sem custo desnecessário de LLM.

Síntese do Módulo 12

5 Lições sobre GraphRAG em 2026

  1. 1
    GraphRAG não é universalmente superior
    Em fact retrieval, perde −13.4% para RAG vetorial. Escolha o paradigma pela complexidade da tarefa, não pela novidade da técnica.
  2. 2
    Construção sem LLM é prática e eficaz
    Dependency parsing (SpaCy) alcança 94% da qualidade LLM-based com fração do custo. Para corpora enterprise, é a única opção escalável.
  3. 3
    Hierarquia de etiquetas supera grafos monolíticos
    TagRAG: 14.6× construção mais eficiente, inserção incremental nativa. Evita reconstrução total ao adicionar novos documentos.
  4. 4
    Fusão de estratégias > estratégia única
    ROGRAG: Logic Form + Dual-Level Fuzzy são complementares. Verificação automática com degradação inteligente garante cobertura sem custo excessivo.
  5. 5
    RRF com embeddings multi-granulares é o padrão
    Embeddings separados para entidade, chunk e relação + Reciprocal Rank Fusion consistentemente supera qualquer retrieval isolado.
Guia de Decisão Prático

Quando Usar Grafo vs. Vetor?

🔵
Use RAG Vetorial quando...
  • ✓ Recuperação de fato direto e isolado
  • ✓ Dados temporais ou atualizações frequentes
  • ✓ Latência <100ms é requisito crítico
  • ✓ Corpus sem hierarquias relacionais explícitas
🟡
Use GraphRAG quando...
  • ✓ Multi-hop reasoning é obrigatório
  • ✓ Corpus com hierarquias domain-rich densas
  • ✓ Síntese de contexto global importa
  • ✓ Dependências entre entidades são cruciais
Padrão Recomendado — Abordagem Incremental
Começar com RAG Vetorial. Adicionar camada de grafo (via RRF) apenas quando benchmarks mostrarem gap em tarefas de raciocínio relacional. Usar TagRAG para inserção incremental sem custo de reconstrução.
